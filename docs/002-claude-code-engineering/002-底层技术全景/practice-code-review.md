# 实践：渐进式构建代码审查自动化

> 配套笔记：[002 · Claude Code 底层技术全景导览](./README.md)
>
> 目标：从零开始，分五个版本渐进式构建一个代码审查自动化流程，每一步都对应笔记中一个核心组件。

---

## 实践背景

笔记里提到 CC 的四层架构和四大组件，光看概念容易晕。这个实践用一个具体场景——**代码审查**——把 Commands、Skills、SubAgents、Hooks、Headless 串起来。

核心理念：**先用最简单的方案跑起来，再逐步引入更高级的组件。** 不要一开始就追求「完美架构」。

---

## 版本零：零配置起步

**场景**：你有一段代码，想让 Claude Code 帮你审查一下。

**做法**：直接在对话中说：

```
帮我审查这段代码，关注以下方面：
- 是否有安全漏洞？
- 是否有性能问题？
- 是否有可读性问题？
```

**效果**：Claude 会直接分析并给出建议。

**局限**：
- 每次都要手动描述审查要求
- 审查标准可能不稳定（不同对话结果差异大）
- 团队其他人没法复用你的审查标准

---

## 版本一：Commands — 固化审查流程

**目标**：把审查标准固化为一个 `/review` 命令，每次审查按统一流程执行。

**步骤**：

1. 创建命令文件 `.claude/commands/review.md`：

```markdown
请对当前修改的文件进行代码审查，按以下流程执行：

## 审查流程

1. **安全检查**
   - 检查是否有 SQL 注入、XSS、CSRF 等安全漏洞
   - 检查敏感信息是否被硬编码
   - 检查用户输入是否有校验

2. **性能检查**
   - 检查是否有不必要的循环嵌套
   - 检查数据库查询是否有 N+1 问题
   - 检查是否有未释放的资源

3. **可读性检查**
   - 检查变量和函数命名是否清晰
   - 检查函数是否过长（建议不超过 50 行）
   - 检查是否有重复代码

## 输出格式

生成一份审查报告，包含：
- 🔴 严重问题（必须修复）
- 🟡 建议改进（建议修复）
- 🟢 做得好的地方
```

2. 使用：

```
/review
```

💡 **这一步的价值**：审查标准从「脑子里的印象」变成了「可复用的文档」，团队可以复用。

---

## 版本二：Skills — 让 AI 自动判断何时审查

**目标**：不需要手动敲 `/review`，当你提到「安全」「审查」「review」时，AI 自动激活代码审查能力。

**不同于 Commands 的关键点**：
- Commands = 你手动触发，100% 确定会执行
- Skills = AI 根据语义自动判断，可能触发也可能不触发

**步骤**：

1. 创建 Skill 文件 `.claude/skills/code-review/SKILL.md`：

```markdown
---
name: code-review
description: 代码审查技能。当用户提到代码审查、安全检查、代码质量、review 等场景时自动激活。
---

# Code Review Skill

## 触发条件（AI 自动判断）
当用户的消息满足以下条件时激活：
- 提到了「审查」「review」「安全检查」「代码质量」等关键词
- 请求对某段代码进行评估或检查
- 询问代码是否存在问题

## 审查规则

### 安全审查（优先级最高）
- SQL 注入、XSS、CSRF 漏洞
- 敏感信息硬编码
- 认证和授权逻辑缺陷
- 用户输入校验缺失

### 性能审查
- 不必要的循环嵌套
- N+1 查询问题
- 未释放的资源（连接、文件句柄）

### 可读性审查
- 命名规范
- 函数长度（> 50 行需标记）
- 代码重复

## 输出格式
生成结构化审查报告，使用 Emoji 标记严重程度。
```

2. 使用（无需手动敲命令）：

```
帮我看看这段代码有没有安全问题
```

AI 自动识别到「安全问题」关键词，激活 `code-review` Skill。

💡 **这一步的价值**：从「手动触发」升级为「语义触发」。但审查结果仍然和对话混在一起，上下文会膨胀。

---

## 版本三：SubAgents — 审批查结果不污染主对话

**目标**：把审查任务丢给专门的 SubAgent 执行，主对话只需接收审查报告。

**为什么需要这一步**：
- Skills 审查后，所有中间分析过程都留在主对话里，对话越长模型注意力越分散
- SubAgent 在独立上下文中完成审查，只把最终报告带回来

**步骤**：

1. 创建 SubAgent 文件 `.claude/subagents/code-reviewer.agent.md`：

```markdown
---
name: code-reviewer
description: 代码审查子代理，负责独立执行代码审查任务
tools: Read, Grep, Glob
---

# Code Reviewer Agent

## 职责
你是专门的代码审查员，只负责审查代码，不负责修改代码。

## 工作流程
1. 读取用户指定的文件或目录
2. 按以下维度逐项审查：
   - 安全性
   - 性能
   - 可读性
   - 最佳实践
3. 生成审查报告，包含问题列表和修复建议
4. 返回报告给主 Agent

## 约束
- 你只有只读权限（Read, Grep, Glob）
- 不要尝试修改任何文件
- 不要执行任何命令
```

2. 使用时：

```
帮我审查 src/ 目录下的代码，用 code-reviewer agent 来做
```

Claude Code 会显示「正在调用 code-reviewer agent」，审查完成后返回报告。主对话上下文保持干净。

💡 **这一步的价值**：上下文隔离 + 权限分离。审查员的上下文是独立的，审查过程产生的噪声不会污染主对话。

---

## 版本四：Hooks + CI — 全自动化

**目标**：在 CI/CD 中自动触发代码审查，发现问题自动评论 PR。

这是版本三的升级——把审查嵌入到自动化流水线中：

**步骤**（概览）：

1. 创建 Hook 文件 `.claude/hooks/pre-commit-review.sh`：

```bash
#!/bin/bash
# 在 commit 前自动触发代码审查
claude --headless "用 code-reviewer agent 审查即将提交的代码"
```

2. 在 GitHub Actions 中集成：

```yaml
name: AI Code Review
on: [pull_request]
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: AI Review
        run: claude --headless "Review the changed files in this PR"
```

> ⚠️ Headless 和 Hooks 的详细配置会在后续课程中展开，这里只是展示演进方向。

## 实践总结

```
版本零   直接对话          "帮我审查这段代码"
   ↓     零配置，立刻能用
版本一   Commands          标准化审查流程，可复用
   ↓     手动触发 /review
版本二   Skills            AI 自动识别审查场景
   ↓     语义触发
版本三   SubAgents         上下文隔离，只拿结果
   ↓     独立执行
版本四   Hooks + CI        全自动化，无人值守
```

**关键原则**：
- **不是一步到位**：从最简单的方案开始，遇到痛点再升级
- **每个版本解决一个具体问题**：零→一 解决标准不统一，一→二 解决手动触发繁琐，二→三 解决上下文膨胀，三→四 解决人工介入
- **每一步都能独立工作**：不是说必须要做到版本四才算完，版本一就已经有价值了

💡 这个渐进式演进本身就是笔记里讲的「从使用者到驾驭者」的路线图。
